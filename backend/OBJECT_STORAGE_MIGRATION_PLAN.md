# Object Storage Migration Plan

这份文档的目标，是把当前“本地 `uploads/` 目录存 PDF”的做法，逐步迁移为“对象存储保存 PDF”。

它不是直接改代码，而是先把迁移边界、步骤和风险讲清楚，避免你到时候一边改一边猜。

## 1. 为什么要迁移对象存储

当前版本的图书 PDF 文件保存在后端本地目录里。

这在本地开发时很方便，但上线后会有几个现实问题：

1. 平台重启后，本地文件可能丢失
2. 多实例部署时，不同实例的本地文件不会自动同步
3. 数据库恢复，不等于 PDF 文件一定还在

所以如果你准备长期运行，图书文件最终应该迁移到对象存储。

## 2. 当前代码里哪些地方和本地文件强绑定

当前最关键的绑定点有两个：

### 后端上传逻辑

在 [backend/controllers/books.js](backend/controllers/books.js) 里，当前上传使用的是：

- `multer.diskStorage()`
- 目标目录是 `uploads/`

### 前端资源访问逻辑

在 [PDF-Reader_frontend/src/utils/http.js](PDF-Reader_frontend/src/utils/http.js) 里，当前 `buildAssetUrl()` 默认把资源路径拼到后端 API 地址上。

这意味着当前默认思路是：

- PDF 文件和后端 API 来自同一个服务地址

而迁移到对象存储后，这个前提就不再成立。

## 3. 迁移后的目标结构

推荐的目标结构是：

1. MongoDB 继续保存图书元数据
2. 对象存储保存 PDF 文件本体
3. 后端返回可访问的文件地址或对象键
4. 前端直接访问对象存储 URL 或签名 URL

你可以把它理解成：

- 数据库管“这本书是谁、叫什么、属于谁”
- 对象存储管“PDF 文件本体放在哪”

## 4. 建议新增的数据字段思路

当前 `Book` 里和本地磁盘强绑定的字段有：

- `filename`
- `path`

迁移后建议保留兼容思路，但逐步引入更明确的对象存储字段，例如：

- `storageProvider`
- `objectKey`
- `fileUrl`

这样以后你一眼就能看出：

- 文件是本地的
- 还是对象存储的

## 5. 推荐的迁移顺序

### Phase 1：只做数据结构兼容

先让后端模型和返回结构支持对象存储字段，但暂时不切上传逻辑。

目标：

- 不影响当前本地联调
- 为后续迁移预留字段

### Phase 2：上传改走对象存储

把管理员上传图书时的文件保存目标，从本地 `uploads/` 改成对象存储。

目标：

- 新上传的图书走对象存储
- 老图书暂时保留兼容

### Phase 3：阅读页改为读取对象存储地址

让阅读页优先使用对象存储地址打开 PDF，而不是默认拼后端静态目录。

### Phase 4：清理本地文件依赖

等所有图书都迁移完成后，再逐步去掉本地 `uploads/` 的主路径依赖。

## 6. 对你这个项目最适合的接入方式

如果你是个人开发者，优先考虑下面这种简单方案：

1. 后端上传成功后，把文件写入对象存储
2. 后端把对象存储 URL 保存到数据库
3. 前端阅读时直接用这个 URL 打开 PDF

这是最容易理解、最少绕弯的方案。

不建议你一开始就上特别复杂的“分片上传 + 回调签名 + 多区域桶”方案。

## 7. 新手最容易踩的坑

### 坑 1：只改上传，不改阅读地址

如果文件已经不在后端本地目录里，但前端还在拼 `/uploads/...`，阅读页就会直接失败。

### 坑 2：数据库只存 URL，不存对象键

如果未来你要迁移域名、换桶、加签名，没有对象键会很被动。

### 坑 3：以为对象存储迁移后，数据库自动更安全

不是。

对象存储解决的是“文件持久化和文件访问”问题，不是数据库权限问题。

### 坑 4：一次性强迁所有旧数据

对当前阶段，不建议先写复杂迁移脚本再开始部署。

更稳妥的方式通常是：

- 先兼容新旧两种文件来源
- 再逐步迁移旧数据

## 8. 当前最推荐的现实做法

你现在最适合的节奏是：

1. 当前先用托管 Node + Atlas 跑起来
2. 文档和部署流程稳定后
3. 再单独开一个小任务做对象存储迁移

这样不会把“上线问题”和“文件存储架构改造”混在一起。

## 9. 迁移完成后的收益

如果后面迁移成功，你会立刻得到这些好处：

1. 平台重启不再导致 PDF 丢失
2. 多实例部署更容易
3. 更适合后续评估 Serverless
4. 数据恢复能力更完整

这也是为什么对象存储迁移是当前路线里的第二阶段重点。
