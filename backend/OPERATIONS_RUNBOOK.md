# Operations Runbook

这份文档是给当前项目上线后的日常运行维护准备的。目标不是做企业级值班手册，而是让个人开发者在遇到问题时，知道先看哪里、怎么判断、怎么恢复。

## 1. 你上线后最应该先学会看的 3 样东西

对于这个项目，日常排查先看：

1. 后端健康检查
2. 平台日志
3. MongoDB Atlas 数据是否正常

不要一上来就怀疑“代码全坏了”。

## 2. 健康检查怎么用

后端已经提供：

- `GET /health`
- `GET /api/health`

如果这两个接口都不通，优先说明：

1. 服务没有正常启动
2. 域名没有正确指向服务
3. 平台部署失败

## 3. 日志里先看什么

上线后最先关注这些日志关键词：

- `MongoDB connected`
- `MongoDB connection error`
- `Register error`
- `Login error`
- `Create book error`
- `Redeem error`
- `Update reading state error`

你可以把它理解成：

- 连接错误先看数据库
- 登录错误先看账号、token、限流
- 上传错误先看文件、磁盘、权限
- 阅读错误先看书架关系和图书状态

## 4. 什么时候应该重启服务

出现下面这些情况时，可以优先尝试平台重启：

1. 健康检查偶发失败
2. 部署后环境变量刚改完，但服务状态不稳定
3. 平台实例异常但代码没有改动

不建议把“重启”当成万能修复。重启前至少先看最近日志有没有明显报错。

## 5. 最常见问题的排查顺序

### 场景 A：前端页面能打开，但接口全部失败

先检查：

1. 后端 `/health` 是否正常
2. `FRONTEND_URL` 是否写成前端真实域名
3. 前端是否还在请求本地地址

### 场景 B：后端启动了，但数据库操作失败

先检查：

1. `MONGO_URI` 是否正确
2. Atlas 白名单是否允许当前平台访问
3. 数据库用户权限是否正确

### 场景 C：管理员上传成功后，文件访问失败

先检查：

1. 平台本地磁盘是否保留了 `uploads` 文件
2. 服务是否发生过重启或实例迁移
3. 当前平台是否适合长期依赖本地文件

### 场景 D：普通用户登录总失败

先检查：

1. 手机号或邮箱是否输入正确
2. 是否触发认证限流
3. token 是否已过期

## 6. 告警至少做哪几类

个人项目不需要一开始就上复杂监控，但最少建议关注：

1. 服务不可用告警：健康检查失败
2. 数据库连接异常告警
3. 部署失败告警

如果平台原生支持邮件通知或 Webhook，先把这三类打通就够了。

## 7. 备份与恢复要怎么理解

你现在最需要防的，不一定是黑客，而更可能是：

1. 误删数据
2. 测试时把生产数据改坏
3. 平台重启后本地文件丢失

所以备份和恢复是必须有说明的。

## 8. 数据库恢复最低要求

对于当前项目，最低要求是：

1. 你知道 Atlas 最近一次备份在哪
2. 你知道如何恢复最近一份快照
3. 你知道恢复后要重新验证哪些业务链路

恢复后至少重新验证：

1. 登录
2. 管理员图书管理
3. 兑换码记录
4. 书架阅读状态

## 9. 关于文件恢复的现实提醒

当前图书 PDF 还在后端本地 `uploads/` 目录，不在对象存储。

这意味着：

1. 数据库恢复，不等于文件一定恢复
2. 即使 MongoDB 里的图书记录还在，PDF 文件本体也可能丢失

所以当前版本的“恢复能力”并不完整，这也是后续必须迁移对象存储的重要原因。

## 10. 一份最简单的日常检查表

你每次部署完成后，最少做这几件事：

1. 打开 `/health`
2. 看平台日志是否出现 `MongoDB connected`
3. 登录一个普通用户
4. 登录一个管理员
5. 打开书架页

如果这 5 步都正常，说明这次部署大概率没有把系统打坏。

## 11. 给个人项目的运维建议

你现在不需要追求复杂运维平台。

最实用的原则只有 3 条：

1. 有健康检查
2. 有日志
3. 有备份说明

只要这 3 件事到位，很多问题你都能自己定位，而不会陷入“系统坏了但完全不知道从哪看”的状态。
