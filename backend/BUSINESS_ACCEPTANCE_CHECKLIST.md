# Business Acceptance Checklist

这份文档用于帮助你完成当前项目的整条业务链路验收。它不是给资深后端写的，而是给第一次自己做联调和验收的人写的。

你只要按顺序做，不要跳步，就能判断当前系统是不是已经基本可用了。

## 1. 验收前准备

开始前请先确认：

1. 后端已经启动成功
2. MongoDB 已连接成功
3. 前端可以正常打开
4. 你知道当前前端请求的是本地后端还是线上后端

如果你是本地联调，通常地址类似：

- 前端：`http://localhost:3000` 或 uni-app H5 本地地址
- 后端：`http://127.0.0.1:5000`

如果你是线上联调，通常地址类似：

- 前端：`https://reader.example.com`
- 后端：`https://api.example.com`

## 2. 第一步：验证后端服务真的活着

先在浏览器里打开：

```text
http://127.0.0.1:5000/health
```

或者线上环境：

```text
https://api.example.com/health
```

预期结果：

```json
{
  "status": "ok",
  "service": "backend"
}
```

如果这里都不通，后面业务验收先不要继续。

## 3. 第二步：注册一个普通用户

在前端注册一个普通用户，示例信息：

- 用户名：`user_a`
- 手机号：`13800138001`
- 邮箱：`usera@example.com`
- 密码：`123456`

预期结果：

1. 前端提示注册成功
2. 数据库 `users` 集合出现一条新数据
3. 这条用户的 `role` 默认是 `user`

## 4. 第三步：验证普通用户可以登录

分别做两次登录测试：

### 方式 A：手机号登录

- `account=13800138001`
- `password=123456`

### 方式 B：邮箱登录

- `account=usera@example.com`
- `password=123456`

预期结果：

1. 两种方式都能登录成功
2. 前端能拿到 token
3. 前端进入登录后的页面

## 5. 第四步：把另一个用户提权成管理员

再注册一个新用户，示例：

- 用户名：`admin_a`
- 手机号：`13800138002`
- 邮箱：`admina@example.com`
- 密码：`123456`

然后进入 `mongosh`：

```javascript
use pdf-reader
db.users.updateOne(
  { phone: '13800138002' },
  { $set: { role: 'admin' } }
)
```

预期结果：

1. 数据库里该用户的 `role` 变成 `admin`
2. 用这个账号重新登录后，前端出现管理员菜单

## 6. 第五步：管理员上传一本图书

使用管理员账号登录后，进入图书管理页，上传一本 PDF。

建议填写：

- 标题：`测试图书 A`
- 作者：`测试作者`
- 分类：`测试分类`

预期结果：

1. 前端提示上传成功
2. `books` 集合新增一条图书数据
3. `backend/uploads/` 或部署平台对应目录里存在新 PDF 文件

## 7. 第六步：管理员为这本书生成兑换码

在图书管理页，对刚上传的图书生成兑换码，例如生成 3 个。

预期结果：

1. 前端显示新生成的兑换码
2. `redeemcodes` 集合新增多条记录
3. 这些记录的 `status` 应为 `unused`

## 8. 第七步：普通用户兑换图书

切回普通用户账号，用一个未使用兑换码进行兑换。

预期结果：

1. 前端提示兑换成功
2. 图书出现在书架首页
3. `bookshelfitems` 集合新增一条记录
4. 对应兑换码状态变成 `used`

## 9. 第八步：进入阅读页并同步进度

普通用户从书架进入阅读页，至少做以下动作：

1. 打开图书
2. 翻页
3. 添加一个书签

预期结果：

1. 阅读页能正常打开 PDF
2. 再返回书架后，最近阅读有内容
3. 数据库中对应 `bookshelfitems` 记录的以下字段发生变化：

- `progress`
- `lastPage`
- `lastReadAt`
- `bookmarks`

## 10. 第九步：验证普通用户不能做管理员操作

用普通用户账号检查：

1. 前端看不到管理员上传入口
2. 前端看不到兑换码管理入口
3. 即使手工请求管理员接口，也应返回权限不足

预期结果：

- 普通用户只能做注册、登录、兑换、书架、阅读相关操作

## 11. 第十步：验证图书下架规则

管理员把某本图书状态改为 `inactive`。

然后验证两件事：

1. 管理员不能再为这本书生成新兑换码
2. 普通用户不能再用未使用兑换码完成兑换

预期结果：

- 系统返回“不可兑换”或“不可生成兑换码”的错误

如果该书之前已经被某普通用户兑换过，还要再确认：

- 已持有用户仍然可以继续阅读

## 12. 第十一步：验证移出书架再兑换

普通用户把已经拥有的书从书架移除。

然后再用新的兑换码兑换同一本书。

预期结果：

1. 图书可以重新回到书架
2. 阅读进度不应继承旧值
3. 书签不应自动恢复旧值

## 13. 第十二步：验证最近 200 条兑换记录

管理员打开兑换记录页。

预期结果：

1. 能看到最近兑换记录
2. 记录里至少包含：

- 兑换码
- 使用时间
- 兑换的图书
- 使用的用户

## 14. 只要这几项全部通过，就说明系统达到当前阶段可用

你可以把“当前版本已经基本可用”的标准先简化为下面 6 句话：

1. 用户能注册和登录
2. 管理员能上传书
3. 管理员能发兑换码
4. 普通用户能兑换书
5. 普通用户能阅读并保存进度
6. 权限边界没有明显错乱

## 15. 如果你卡住了，优先按这个顺序判断问题属于哪一类

1. 服务没启动
2. 数据库没连上
3. 前端请求地址不对
4. 权限配置不对
5. 业务逻辑本身有 bug

这样你就不会把所有问题都混在一起看。
