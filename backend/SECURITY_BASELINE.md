# Security Baseline

这份文档用于说明当前项目已经落地的最小安全基线，以及你后续部署时还要继续确认的点。

## 1. 当前已经落地的安全措施

当前后端已经具备以下最小安全基线：

1. 生产环境 CORS 不再默认放开所有来源，而是依赖 `FRONTEND_URL`
2. 已启用基础安全响应头中间件 `helmet`
3. 已关闭 `X-Powered-By` 头，减少框架暴露信息
4. 认证接口已增加限流，防止登录和注册被短时间暴力请求
5. JSON 和表单请求体已增加体积限制，避免异常大请求直接打进应用层

## 2. 本次安全限制的具体含义

### CORS

生产环境下：

- 只允许 `FRONTEND_URL` 指定的前端来源访问

开发环境下：

- 为了本地联调方便，仍然允许 `*`

### Helmet

`helmet` 会帮助你自动加上一批基础安全响应头。你现在不需要理解每个头的细节，只要知道它是在帮你提高浏览器访问的默认安全基线。

### 登录限流

当前对以下接口增加了限流：

- `POST /api/auth/register`
- `POST /api/auth/login`

规则是：

- 15 分钟内最多 20 次请求

超出后会返回：

```json
{
  "message": "Too many authentication requests, please try again later"
}
```

### 请求体大小限制

当前设置为：

- JSON 请求体最大 `2mb`
- URL encoded 请求体最大 `2mb`

这不会影响 PDF 上传，因为 PDF 上传走的是 `multer` 文件上传流程。

## 3. 你部署时还要继续做的事情

虽然代码里已经有最小安全基线，但这不代表部署后就自动安全。你还需要继续确认：

1. `JWT_SECRET` 已替换为强随机字符串
2. `MONGO_URI` 使用专用应用账号，不是管理员账号
3. `FRONTEND_URL` 写的是前端真实域名
4. 前后端都使用 HTTPS
5. 数据库账号权限是最小化的

## 4. 当前还没有做的安全增强

下面这些不是本次必须项，但后续可以再做：

1. 更细粒度的上传频率限制
2. 管理员接口审计日志
3. 对象存储签名 URL
4. 更严格的内容安全策略
5. 登录失败次数统计和告警

## 5. 关于 Node 版本的提醒

当前仓库里的依赖链已经对 Node 版本提出了更高要求，特别是 `mongoose@9` 更偏向 Node 20 环境。

所以部署平台建议优先选择：

- Node 20 LTS

如果你本地还是 Node 18，项目可能还能跑，但不建议把 Node 18 当成长期生产标准。
