# Serverless Evaluation Guide

这份文档不是让你现在立刻切 Serverless，而是告诉你：在什么时候、满足哪些条件之后，再去评估这件事才划算。

## 1. 先说结论

对当前项目来说，**现在先上托管 Node 服务是更稳的**。

只有在下面这些前提基本满足后，再去评估 Serverless 才比较合理：

1. 图书文件已经迁移到对象存储
2. 前端不再依赖后端本地 `uploads/`
3. 后端部署流程已经稳定
4. 业务闭环已经验证过多轮

## 2. 为什么不能太早切 Serverless

如果现在直接切，你会把两个问题混在一起：

1. 业务有没有跑通
2. 架构是否适合函数化

这对个人项目很不划算，因为你会很难分清到底是：

- 代码逻辑问题
- 文件存储问题
- 平台限制问题
- 冷启动问题

## 3. 当前项目里哪些部分天然适合 Serverless

下面这些接口天然更适合函数化：

1. 注册
2. 登录
3. 兑换码生成
4. 兑换码兑换
5. 书架查询
6. 阅读进度同步

这些接口的共同特点是：

- 请求短
- 业务边界清晰
- 不强依赖本地磁盘

## 4. 当前项目里哪些部分不适合直接原样函数化

当前最不适合直接原样迁移的，是和文件强绑定的部分：

1. 图书上传
2. 本地静态文件访问
3. 依赖 `uploads/` 的文件读取

所以你会发现，Serverless 是否可行，关键根本不在 JWT，而在文件存储链路。

## 5. 什么时候可以开始认真评估

如果下面 5 个问题你都能回答“是”，就可以认真进入 Serverless 评估阶段：

1. 新上传图书已经走对象存储了吗？
2. 阅读页已经不再默认拼 `/uploads/...` 了吗？
3. 后端健康检查、日志、备份已经稳定了吗？
4. 当前托管 Node 方案已经跑过一段时间且业务稳定了吗？
5. 你现在遇到的痛点，真的是“常驻服务成本”而不是“文件存储风险”吗？

如果这 5 个问题里还有多个回答不了，就不建议现在切。

## 6. 评估时最重要的 4 个维度

### 1. 冷启动

要评估：

- 登录、兑换、书架查询这类接口对冷启动是否敏感

### 2. MongoDB 连接复用

要评估：

- 函数平台里如何复用 MongoDB 连接
- 高并发或频繁冷启动时连接是否稳定

### 3. 调试成本

要评估：

- 现在排查问题是不是已经很费劲
- 切成函数后会不会更难定位问题

### 4. 真正能省下什么

要评估：

- 你是能明显节省成本
- 还是只是把问题从“服务运维”换成“函数平台调试”

## 7. 给个人项目最现实的判断标准

你可以把“要不要切 Serverless”先简化成下面 3 个问题：

1. 现在常驻 Node 服务的成本是不是已经明显不值了？
2. 文件存储是不是已经和后端彻底解耦了？
3. 你愿不愿意接受更高的调试复杂度，换取更轻的运行形态？

如果第 2 条还没做到，就不要急着看第 1、3 条。

## 8. 最适合你的推进顺序

对你当前项目，最合理的顺序是：

1. 托管 Node + Atlas 跑稳定
2. 完成对象存储迁移
3. 只挑部分纯业务接口试评估函数化
4. 再决定是否全面迁移

也就是说，**不要一口气把整个后端原样搬到 Serverless。**

## 9. 当前阶段的明确建议

当前阶段，你更应该把 Serverless 当成：

- 后续可选路线

而不是：

- 现在必须立刻执行的目标

这样更符合你这个项目的投入产出比。
